# TDD × AI 协作工作流

本文档描述本仓库的开发方法论:如何用 AI 做结对程序员,同时保持严格的 Test-Driven Development。

## 角色分工

| | 人(Zhien) | AI |
| --- | --- | --- |
| 需求 | 写验收标准(Given/When/Then) | 提出边界情况补充 |
| 红 | **审查测试是否真的表达需求** | 只写测试,不写实现 |
| 绿 | 审查实现 | 写让测试通过的最小实现 |
| 重构 | 决定是否采纳 | 提出重构建议并执行 |

核心原则:**测试即规格**。测试是人审查过的合同,实现只是履行合同。所以 AI 绝不允许同时生成测试和实现——那样合同就是实现方自己写的。

## 单个功能的完整循环

以"整单折扣按净额比例分摊"为例:

1. **验收标准**(人写):
   > Given 一张发票有商品 A(净额 10€)和 B(净额 30€),整单折扣 4€
   > When 执行折扣分摊
   > Then A 分摊 1€、B 分摊 3€,且分摊总和精确等于 4€(尾差用最大余数法)

2. **红**:AI 生成 `allocateDiscount.test.ts`,含正常路径 + 边界(单商品、折扣为 0、除不尽的尾差、金额为整数分)。人审查:测试是否漏了场景?断言是否太宽松?→ 跑测试,确认全红。
   → `git commit -m "test: core: 整单折扣按净额比例分摊"`

3. **绿**:AI 写最小实现,只求通过。
   → `git commit -m "feat: core: 实现折扣分摊"`

4. **重构**:测试保持绿,整理命名/提取函数。
   → `git commit -m "refactor: core: ..."`

5. **记录**:在 `docs/tdd-log/` 写日志,重点记录 AI 输出中被人否决或修正的地方。

## 测试金字塔

| 层 | 工具 | 内容 | 占比 |
| --- | --- | --- | --- |
| 单元 | Vitest | `packages/core` 纯函数,用真实发票数据做 fixtures | ~70% |
| 组件 | Jest + RN Testing Library | 勾选交互、校对表格、汇总展示 | ~20% |
| 集成/E2E | Vitest + Docker Postgres;Maestro | API 路由/鉴权/share_token 权限;移动端关键路径 | ~10% |
| AI 评测 | 固定发票样本 + 字段断言 | 防 prompt 回归,独立 CI job | — |

## CI

push/PR:lint → typecheck → 单测 → 组件测试;main 分支追加 Maestro 冒烟。全绿才能合并。

## 为什么值得这么做

发票分账的核心是**钱的计算**:税率、折扣分摊、均摊、尾差。错一分钱,汇总就对不上发票总额,朋友之间就要吵架。这类纯逻辑恰好是 TDD 回报率最高的地方——而 AI 时代,测试是人对 AI 产出唯一可规模化的验收手段。
