# ADR 0003: 金额表示 —— 整数分 + 千分位定点,取整四舍五入远离零

日期:2026-07-07 · 状态:已接受

## 背景

Metro 发票含 KG 计重行(数量 3 位小数)与 0.1 分精度的折后有效单价(如 6.488€/kg)。
浮点欧元不可接受(手工 Excel 的浮点汇总已产生 ±2 分级误差);单用整数分又装不下单价精度。

## 决策

| 量 | 表示 | 例 |
| --- | --- | --- |
| 金额(净额/税额/含税) | 整数分(cent) | 5.58€ → 558 |
| 数量 | 整数千分位 | 2.871kg → 2871;2 件 → 2000 |
| 单价 | 整数千分位欧(=0.1 分) | 6.488€ → 6488 |

- 行净额 = round(qtyMilli × unitPriceMilli / 10000) → 分;取整一律**四舍五入、远离零**
  (负数对称,即德国 kaufmännische Rundung),由 `roundHalfAwayFromZero` 统一实现。
- 税率为整数百分比,按国家配置(`DEFAULT_TAX_RATES`:DE A19/B7,NL A21/B9),计算层不硬编码。
- 一切拆分(折扣、均摊、按份认领、税额按家庭分摊)经 `allocateByLargestRemainder`:
  全整数运算,结果之和**精确等于**被拆总额,并列余数按下标靠前优先(确定性)。

## 连带决策:validate() 是差额上报器,不是布尔门

fixture 实测:发票印刷净额无法从转录的(数量×单价)按任何有原则的取整规则精确重算(差 -1 分,
原因是三位小数单价系印刷行总额反推、每行 ±0.5 分损耗,详见 docs/tdd-log/001-m1-core.md)。
故 validate() 返回逐项精确差额(计算−印刷)交上层高亮(PRD A4);`BillItem.printedLineNetCents`
允许 OCR 直接采信印刷行总额,届时可 0 差额。

## 代价与缓解

- 千分位表示的输入解析需守边界:`toMilli` 拒绝 >3 位小数;server 层(M2)在 API 边界统一转换。
- 0.1 分单价×克级数量的乘积在 Number 安全整数范围内(≤ ~10^8),无 BigInt 必要。
