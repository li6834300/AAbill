# ADR 0005:税率从欧委会 TEDB 同步生成,不硬编码、也不运行时调 API

## 状态

已接受(2026-07-22)

## 背景

支持多国后需要一张「国家 → 增值税率」表作为兜底(发票读不出税率时用)。
第一版由我凭记忆写了 30 个国家。拿欧委会 TEDB 的真实数据核对后,**30 国里错了 4 个**:

| 国家 | 我填的 | 实际 |
| --- | --- | --- |
| 罗马尼亚 标准税率 | 19% | **21%**(2025 年上调) |
| 芬兰 低税率 | 14% | 上游为 [10, 13.5] |
| 立陶宛 低税率 | 9% | [5, 12] |
| 卢森堡 低税率 | 3% | 3% 属 super-reduced,reduced 为 [8, 14] |

用户一针见血:「万一某个国家哪一天税率调整了,不就完蛋了吗?」

这类数据的失败模式是**错了不会报错** —— 只会静静算出一个错的分账金额。

## 决策

税率从欧委会 TEDB(经 `vatnode/eu-vat-rates-data`,MIT,每日 07:00 UTC 从 TEDB SOAP API 同步)
拉取,由 `scripts/sync-vat-rates.mjs` 生成 `packages/core/src/vat-rates.generated.ts`,
**生成物提交进仓库**。

三道保险:

1. **过期即失败**:`vat-rates.test.ts` 断言数据抓取日期不早于 180 天,超期 CI 红。
2. **每周定时同步**:CI 定时任务跑同步,有变动自动开 PR 待人审(税率变动直接改金额,必须有人看)。
3. **发票优先**:识别发票时优先读发票上印的实际税率百分比,国家表只是读不出时的兜底。

## 为什么不在运行时调 API

1. **算钱路径不该依赖第三方可用性** —— 上游挂了不能让结算跟着挂;
2. **税率若在用户结算途中变动,同一张账单会前后不一致**;
3. 生成物入库后,每次税率变动都是一个可审查、可回滚、有 git 历史的 PR,
   而不是某天悄悄换了个数字。

## 为什么多档低税率不自动选

TEDB 给出的法国低税率是 `[0.9, 1.05, 2.1, 5.5, 8.5, 10, 13]`(含海外省与科西嘉特别税率)。
「食品适用哪一档」从数据里推导不出来 —— 取最低会选中 0.9%,算出来的钱是错的。

故:低税率只有一档时(德国 7%、荷兰 9%)才允许兜底自动采用;
多档时 `taxRates` 留 `null`,由用户对照发票选,UI 列出该国真实档位。
`isReducedRateAmbiguous()` 封装这个判断。

## 代价

- 引入一个第三方仓库作为 TEDB 的中转(直连 TEDB 是 SOAP,成本高)。
  信任假设已在生成物头部与本 ADR 中写明;若该仓库失效,换源只需改 `SOURCE_URL` 与解析逻辑。
- 国名只有英文,中文名暂放在 mobile 的 UI 层,做 i18n 时并入语言包。
