# 012 税率以发票为准,税制改常驻下拉

011 上线后用户提了三条,每条都成立:

1. 「你应该先让我上传了发票再有税制识别或者选择的选项啊。现在是我一上来你就跟我说税制未识别,而且这么显眼干啥」
2. 「我用 metro 的发票测试。这么明显的发票它居然识别不出来是德国税制?是不是 prompt 有问题?」
3. 「那个选择税制应该是个 dropdown 吧。你应该要列出所有国家的税制」

## 识别失败的根因(不是模型的锅)

把用户那张 PDF 的文本流解出来(自定义字体编码,字符偏移 +30),看到:

| 位置 | 内容 |
| --- | --- |
| 卖方抬头 | `METRO Deutschland GmbH`、`USt-IdNr.: DE184048859`、`40235 Düsseldorf` |
| **买方** | `Kasteel Beusdaelplein 11A`、`NL 6222 WD` |

跨境采购:德国门店开票给荷兰客户,发票上 DE 与 NL 并存。而 011 的 prompt 写的是
「依据发票上的**地址**/邮编、语言、税号格式判断」—— **没说是谁的地址**。
模型看到买卖双方国家冲突,又被我叮嘱过「证据不足就返回 UNKNOWN,不要猜」,
于是老老实实回了 UNKNOWN。是 prompt 的规格漏洞,不是模型能力问题。

修法:prompt 明确「只看开票方/门店一侧,**忽略买方与收货地址**,
两者不同时一律以卖方为准,不要因此返回 UNKNOWN」。并加了断言锁死这条规则的回归测试。

诊断全程没有调用一次 OpenAI —— 本地解 PDF 文本就拿到了决定性证据。

## 更重要的一处改法:税率不该来自硬编码表

用户要「列出所有国家的税制」,直觉做法是硬编码一张欧洲税率表。但:

- 税率会变(芬兰 2024、爱沙尼亚 2025 都调过标准税率);
- A/B 两档模型套不住西班牙 4%、爱尔兰与英国食品 0% 这类超低/零税率;
- 表填错了**不会报错**,只会算出一个错的分账。

而欧盟发票法定必须印税率明细 —— 那张 Metro 发票底部就写着 19,00% 与 7,00%。
**发票上印的才是当时当地真实收的税**,比任何表都权威。

与用户确认后定为:**读发票的实际税率 + 国家表兜底**。
国家从此只是标签与兜底来源,`bill.taxRates` 才是算钱用的东西,
校验/结算/锁定的 409 守卫也改看 `taxRates` 而非 `taxCountry`
—— 国家读不出但税率读得出时,照样能算钱。

## 表示法:整数百分比 → 整数基点(bp)

法国 5.5%、瑞士 8.1%、芬兰 25.5%、爱尔兰 13.5% 都不是整数。
整数百分比装不下,浮点百分比又会把误差带进分账取整(违反 ADR 0003 的整数原则)。
故 `TaxRates` 改为整数基点:19% = 1900,5.5% = 550,`vatCents` 全程整数运算。

## AI 输出被人否决/修正的点

- 我用 `git add -A` 把实现混进了 `test:` 提交,违反「测试与实现分开提交」。
  发现后 `reset --soft` 拆成两个提交。**这是我自己的流程失误,不是工具问题。**
- 我最初想直接硬编码 30 国税率表交差。停下来想了一下:我凭记忆填的税率错了不会有任何信号,
  而这个项目最不能接受的失败模式就是「算错钱却不报错」。于是把这个取舍交给用户定,
  并在表上明写「这是兜底,不是权威」与数据截止日期。

## 已知局限

`DEFAULT_TAX_RATES` 的 30 国数据截至 2026-07,由我按记忆填写,**芬兰、爱沙尼亚、
斯洛伐克近年调整过税率,值得人工复核**。缓解措施有三层:发票税率优先于表、
校验页拿印刷合计交叉核对(税率用错会显示为差额)、表内注明数据日期。
