# 010: 识别质量排查(多页发票漏识别 + Metro 折扣列)

日期:2026-07-08 · 分支:m13-google-fix

## 现象

用户传真实 Metro 发票(730 €,多页),识别结果 validate 报 `净额差 -450.53 €`。
用户反馈"ChatGPT 识别得很清楚"。

## 排查(用真实 PDF + 我们的管线实跑)

1. gpt-4o 只返回 **19 行、行净额之和 226.62**,但模型自己读到的净额是 **677.15**。
2. pdftotext 抽出发票全文:第 1 页末尾是 `SEITENSUMME: 226,62` 然后 `SEITE: 2`。
   → **模型只提取了第 1 页,后面几页全漏**;却从末页读到了最终总计。validate() 正确抓出了这个漏识别。
3. 换 **gpt-4.1 / gpt-5.1**:都提取了 **40 行**,多页问题解决。但行净额之和 722.65 > 净额 677.15。
4. 722.65 是 GESAMT 列(折扣前)之和;Metro 的 `*` 折扣行真实价在 **INT KD PREIS**(最后一列)。
   我们旧提示词让模型"抄印刷行总额(Wert 列)",抄成了折扣前价。

## 修复

- **默认模型 gpt-4o → gpt-4.1**(`OPENAI_MODEL` 可覆盖):多页完整提取,快(~22s)且比 gpt-5.1 省。
- **提示词重写**:①必须逐页提取全部行、不在 SEITENSUMME 处停;②跳过页眉/章节/LOT/MHD 附加行;
  ③`unitPriceNet = INT KD PREIS`(折后净单价),`lineNet = qty × unitPriceNet` 自己算,不抄 GESAMT;
  ④自检:Σ lineNet 应约等于 totals.net。

## 验证(真实发票)

gpt-4.1 + 新提示词:40 行,**行净额之和 677.21 vs 净额 677.15,差 0.06 €**(四舍五入尾差)。
从差 450 € 到差 6 分,validate 基本对上,可用。

## 学到的

1. **validate() 是这次的英雄**:它把"漏识别"量化成一个显眼的差额,否则用户会以为识别对了。
   这正是当初把 validate 设计成"差额上报器"而非布尔门的价值(ADR 0003 / tdd-log 001)。
2. 模型选型对"密集多页表格提取"影响巨大;gpt-4o 会在长文档上偷懒,gpt-4.1 明显更实。
3. 领域细节(Metro INT KD 折扣列)必须写进提示词;模型不会自己猜对该用哪一列。
