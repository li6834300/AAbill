## 关键发现:fixture 的 -1 分「转录伪差」(本次最重要的规格决策,请人重点抽查)

fixture notes 断言「validate() 复现行级四舍五入规则即可对上印刷合计」。**实测不成立**:按行取整重算
Σ净额=517.20,发票印 517.21,差 -1 分;且无任何有原则的取整规则(half-up/half-even/逐级进位/含税行取整)能凑出 +1。
证据:同一商品(RINDER-FALSCHES FILET)单价出现 12.288/12.290/12.291/12.293 四种——三位小数「有效单价」
是从印刷行总额**反推**的,每行有 ±0.5 分损耗;歧义行有 9 条,无法唯一定位差异行。

处理:
- 取整规则定为「行净额=round(qty×单价) 远离零到分」,两个税类税额与印刷值**完全吻合**,净额/含税差 -1 分
- validate() 语义定为**精确差额上报**(计算−印刷),不做布尔门、不设容差——这正好匹配 PRD A4
  「不匹配时高亮差额、引导人工排查」的产品流程
- BillItem 增加可选 `printedLineNetCents`:OCR 能读到印刷行总额时优先采用(M2 起 AI 识别应输出行总额,
  而不仅是单价×数量),届时可实现 0 差额对账
- 集成测试如实断言 `{net:-1, vatA:0, vatB:0, gross:-1}` 并注明原因

## 学到的

真实世界的「golden 数据」自身可能不自洽:发票印刷合计对,但从中转录的字段重算不回去。
核心库的正确姿势不是强行凑数,而是把差额如实暴露给人——这本来就是产品要的行为。
